## 1. Seed, Nonce & Determinism

**Player question:** Can the casino change your outcome after you bet?

Every Plinko drop on Duel is generated from three inputs: server seed, client seed, and nonce. The casino commits to its seed before you bet, you control your own client seed, and the nonce increments automatically. At each row, these inputs are combined via HMAC-SHA256 to produce a single left/right decision. The ball's final slot is the sum of all right-turns across every row.

This section validates whether Duel seed handling meets provably fair standards and whether the implementation is
deterministic and tamper-resistant.

  <h4>1.1 Server seed commitment</h4>
  <p>
    Before any bet is placed, the casino generates a secret server seed and publicly commits to it by displaying its

    revealed value and confirms it matches the pre-committed hash.
  </p>
  <p>
    Result: all <code>152</code> revealed seeds pass
    <code>SHA-256(hexDecode(serverSeed)) = serverSeedHashed</code> with zero mismatches.
  </p>

  <h4>1.2 Commitment linkage (pre-bet verification)</h4>
  <p>
    At the start of each phase (A, B, C), the active <code>server_seed_hashed</code> was captured before betting.
    After the first bet of each phase, the response hash was compared against the pre-captured commitment.
  </p>
  <p>
    Result: all <code>3/3</code> pre-capture commitments match the first bet response hash.
  </p>
  <p>
    Known limitation: explicit pre-bet snapshots were captured at three phase boundaries only. For remaining seed
    pairs, commitment appears first in the first post-rotation bet response. Step 3 verifies hash constancy within each
    seed pair; Step 1 verifies revealed-seed hash correctness.
  </p>

  <h4>1.3 Hash consistency within epoch</h4>
  <p>
    For each of the <code>152</code> seed pairs, <code>server_seed_hashed</code> is checked across all bets in the
    pair. Any intra-pair hash change is a commit-reveal violation.
  </p>
  <p>
    Result: <code>152/152</code> seed pairs show a constant hash across all bets. No mid-session seed switches.
  </p>

  <h4>1.4 Player client seed control</h4>
  <p>
    You can view, modify, or randomize your client seed in the Duel UI before placing bets. The client seed is a direct
    HMAC input. Phase D explicitly tests <code>10</code> distinct client seeds to confirm active usage.
  </p>

  <h4>1.5 Nonce incrementation</h4>
  <p>
    Nonce starts at <code>0</code> and increments by <code>+1</code> per bet under the same server/client seed pair.
    Nonce resets to <code>0</code> only when the player rotates client seed and a new server seed is issued. There is no
    automatic rotation after a fixed number of bets.
  </p>
  <p>
    Result: <code>152</code> seed pairs checked; <code>147</code> show clean nonce sequences. <code>5</code> show a
    capture-retry artifact (informational) with no fairness impact; slot recomputation still passes for captured bets.
  </p>

  <h4>1.6 Deterministic mapping</h4>
  <p>
    Given identical <code>(serverSeed, clientSeed, nonce, rows)</code>, the algorithm always returns the same slot.
    All <code>7,600</code> bets with revealed seeds recompute to the exact <code>final_slot</code>. drand is not
    required for slot computation.
  </p>

  <h4>1.7 Client seed influence</h4>
  <p>
    Recomputing all bets with <code>WRONG_CLIENT_SEED_FOR_AUDIT_TEST</code> changes <code>6,409/7,600</code> slots
    (<code>84.3%</code>). This confirms client seed is a genuine HMAC input.
  </p>
  <p>
    Unchanged outcomes (<code>15.7%</code>) are expected coincidences where row bits happen to match despite different
    client seeds.
  </p>

  <h4>What this means for players</h4>
  <ul>
    <li>The casino cannot change your outcome after you place a bet.</li>
    <li>You contribute your own randomness through the client seed.</li>
    <li>Every bet stays unique because nonce increments per bet.</li>
    <li>Any Plinko result can be independently verified.</li>
    <li>Outcomes stay reproducible over time from disclosed inputs.</li>
  </ul>

## 2. RNG & Entropy Integrity

**Player question:** Is the randomness unbiased, or could it be rigged?

The Duel Plinko RNG uses HMAC-SHA256 with deterministic inputs (serverSeed, clientSeed, nonce, cursor). For each row, the algorithm produces one bit, left (0) or right (1), by taking the first 4 bytes of the HMAC output as uint32 and computing <code>int % 2</code>. Slot distribution follows binomial B(n, 0.5); 2³² mod 2 = 0, so there is zero modulo bias.

This section tests whether the RNG produces unbiased results and whether each bet is isolated from all others.

  <h4>2.1 RNG function implementation</h4>
  <p>
    Duel Plinko uses HMAC-SHA256 with deterministic inputs <code>(serverSeed, clientSeed, nonce, cursor)</code>. For

    <code>int % 2</code>.

  <p><strong>Code implementation</strong></p>
  ```ts
  // Source: src/rng.ts — computeSlot
  export function computeSlot(
    serverSeed: string, clientSeed: string, nonce: number, rows: number
  ): number {
    const key = Buffer.from(serverSeed, "hex");
    let slot = 0;
    for (let cursor = 0; cursor < rows; cursor++) {
      const message = `${clientSeed}:${nonce}:${cursor}`;
      const hmac = crypto.createHmac("sha256", key).update(message).digest("hex");
      const hex4 = hmac.substring(0, 8);
      const int = parseInt(hex4, 16);
      slot += int % 2;
    }
    return slot;
  }
  ```
  <p>
    <strong>Key encoding:</strong> server seed is hex-decoded to a 32-byte buffer.  
    <strong>Message format:</strong> <code>clientSeed:nonce:cursor</code> with integer nonce and zero-based cursor.
  </p>

  <h4>2.3 Modulo bias analysis</h4>
  <p>
    Row direction uses <code>int % 2</code> where <code>int</code> is uint32 in
    <code>[0, 4,294,967,295]</code>. Because <code>2^32 mod 2 = 0</code>, there is no modulo bias.
  </p>
  ```text

  4,294,967,296 / 2 = 2,147,483,648 (exact)
  ```
  <p>

    <code>1/2</code>.
  </p>

## 5. Fairness Integrity Testing

**Player question:** Can anyone cheat the system, players or casino?

This section tests whether any party, player or casino, can predict outcomes, alter results, or gain an unfair
advantage through weaknesses in the provably fair implementation under realistic gameplay conditions.

    Every provably fair system makes implicit guarantees to players. This section tests whether those guarantees hold
    under adversarial conditions by attempting to predict outcomes before betting, alter outcomes after betting, replay
    or manipulate cryptographic inputs, leverage cross-round or cross-user state, and tamper with parameters or
    payouts client-side.
